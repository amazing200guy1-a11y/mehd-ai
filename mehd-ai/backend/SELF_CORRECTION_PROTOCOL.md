# SELF_CORRECTION_PROTOCOL.md

PURPOSE:
This file defines the mandatory self-correction, doctrine revalidation, post-task auditing, and continuous refinement loop for MEHD AI development.

The goal is to ensure:

- no careless implementations
- no unfinished execution
- no hidden regressions
- no doctrine violations
- no architectural decay
- no fake completion states

====================================================
MANDATORY EXECUTION LOOP

BEFORE ANY TASK:

1. Read ALL doctrine files completely.
2. Extract implementation requirements.
3. Analyze risks and architecture impact.
4. Define internal target quality standard.

DURING TASK:

1. Continuously cross-check doctrine alignment.
2. Prevent shortcuts.
3. Prevent technical debt accumulation.
4. Prevent UI/UX degradation.
5. Prevent security regressions.
6. Prevent architecture inconsistency.

AFTER TASK:

1. Re-read ALL doctrine files AGAIN.
2. Audit the implementation line-by-line.
3. Verify compliance against ALL doctrines.
4. Identify weaknesses and hidden regressions.
5. Refine implementation until standards are met.

====================================================
MANDATORY RE-READ FILES

The AI MUST re-read:

- COMPULSORY.md
- PRODUCTION_STANDARDS.md
- RED_TEAM_PROTOCOL.md
- ARCHITECTURE_DISCIPLINE.md
- HISTORICAL_FAILURES.md
- SCALING_FAILURE_PATTERNS.md
- SELF_CORRECTION_PROTOCOL.md

before AND after implementation.

====================================================
POST-TASK AUDIT REQUIREMENTS

After every implementation, the AI must verify:

ARCHITECTURE

- modularity preserved
- separation of concerns preserved
- no giant mixed-responsibility files introduced
- no hidden coupling introduced

UI/UX

- responsive on all device sizes
- no overflow warnings
- no clipped layouts
- no inaccessible controls
- no dead buttons
- no broken navigation
- no fake loading states

SECURITY

- no exposed secrets
- no insecure flows
- no unsafe permissions
- no replay attack vectors
- no dangerous assumptions
- no trust leaks between frontend/backend

PAYMENT ARCHITECTURE

- no accidental native IAP assumptions
- no pricing logic scattered across UI
- no hardcoded subscription logic
- entitlement system preserved

SCALABILITY

- no unnecessary polling
- no dangerous memory usage
- no unbounded loops
- no unsafe concurrency
- no race-condition-prone logic

PERFORMANCE

- no unnecessary rebuilds
- no expensive UI loops
- no blocking operations on main thread
- no dangerous async patterns

PROFESSIONALISM

- UI remains institutional-grade
- UX remains calm and structured
- no “toy app” feeling introduced
- animations remain disciplined
- layouts remain clean

====================================================
QUALITY THRESHOLD SYSTEM

The AI must define an INTERNAL QUALITY TARGET before task completion.

The task is NOT complete if:

- doctrine violations remain
- regressions remain
- hidden bugs remain
- dead interactions remain
- architecture remains messy
- responsiveness issues remain
- maintainability concerns remain
- scalability risks remain
- polish level remains low

The AI must continue refining until:

- production-grade quality is reached
- architectural cleanliness is strong
- doctrine compliance is high
- regression risk is minimized
- UX polish is high
- operational discipline is preserved

====================================================
ANTI-FAKE-COMPLETION RULE

The AI must NEVER:

- claim completion prematurely
- stop after superficial fixes
- ignore edge cases
- ignore hidden instability
- leave partial integrations
- leave placeholder systems
- leave disconnected buttons
- leave inconsistent states

====================================================
FINAL DIRECTIVE

MEHD AI is being engineered as:

- resilient infrastructure
- institutional-grade software
- a scalable operating system
- a disciplined execution platform

The AI must continuously self-correct until:

- quality is high
- architecture is stable
- doctrine compliance is strong
- UX is polished
- scalability is protected
- professionalism is preserved

====================================================
MANDATORY DEEP DOCTRINE INGESTION

The AI MUST slowly and carefully process EVERY doctrine file line-by-line and extract EVERY actionable point, restriction, warning, philosophy, architecture rule, security rule, UX rule, scalability rule, payment rule, anti-regression rule, and implementation requirement.

The AI must NOT:

- skim
- summarize lazily
- partially read
- forget earlier sections
- ignore “small” rules
- apply doctrine selectively

The AI must internally retain and continuously remember ALL extracted doctrine points throughout the ENTIRE task lifecycle.

Before implementation begins, the AI must internally build a complete operational memory of:

- all requirements
- all restrictions
- all engineering standards
- all architecture principles
- all security expectations
- all UX expectations
- all scalability expectations
- all anti-chaos rules

The AI must continuously reference this internal doctrine memory during:

- planning
- coding
- refactoring
- reviewing
- debugging
- testing
- polishing
- final auditing

The AI must actively ensure that NO doctrine point is forgotten, skipped, weakened, violated, or lost during implementation.

Completion is forbidden unless ALL doctrine principles remain respected and enforced simultaneously.
